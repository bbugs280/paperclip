# Non-Git Data Backup & Restore Guide

## Overview

This guide covers backing up and restoring all **non-Git data** in Paperclip — company state, database, configuration, and secrets that are vulnerable to AI-induced corruption.

**Key concern**: Time Machine protects you from accidental deletion, but you want **point-in-time snapshots** before AI agents make modifications.

---

## What's Backed Up

### 🟢 CRITICAL (< 100KB - restore first)
- **Encryption master key** (`~/.paperclip/instances/default/secrets/master.key`)
  - **Why**: Required to decrypt all stored secrets and API keys
  - **Impact**: Without this, backup is unusable
  
- **Configuration** (`~/.paperclip/instances/default/.env`)
  - Database connection, adapter settings, feature flags

### 🔵 DATABASE (226MB uncompressed)
- **PostgreSQL data** (`~/.paperclip/instances/default/db/`)
- **Includes**: All application state
  - Companies, agents, tasks, issues, routines
  - User data, API keys (encrypted), documents
  - Activity logs, comments, decisions
- **Vulnerability**: AI agents can modify, corrupt, delete
- **Recovery impact**: Complete loss without this

### 🟡 COMPANY & AGENT STATE (528KB)
- **Location**: `~/.paperclip/instances/default/companies/`
- **Contents**:
  - Agent instructions & prompts (AI-written)
  - Shared scripts (AI-written shell/Python)
  - Company documents generated by agents
  - Issue documents, editions
- **Vulnerability**: PRIMARY risk area for AI corruption
- **Example**: Agent accidentally overwrites a playbook or shared script

### 🟠 FILE STORAGE (5.2GB)
- **Location**: `~/.paperclip/instances/default/data/`
- **Contents**:
  - User uploaded images, files
  - AI-generated content (documents, exports)
  - Cached data, temporary files
- **Vulnerability**: AI agents create/modify files here
- **Recovery time**: 5-10 min (largest component)

### 🟤 OTHER STATE (5.1MB)
- **Workspaces** (~208KB): Active workspace snapshots
- **Projects** (~4.7MB): Project metadata, custom settings
- **Runtime Services** (~12KB): MCP server state
- **Skills** (~420KB): Learned skill definitions
- **Custom Plugins/Tools**: User-installed adapters, plugins

## Backup Strategy

### Option A: Pre-Agent Run (Immediate)
```bash
# Before running AI agents that might corrupt data
./scripts/backup-non-git-data.sh .backups/pre-build-agent

# Run agent
pnpm paperclipai run build-agent

# If corruption, restore
./scripts/restore-non-git-data.sh .backups/pre-build-agent-*
```
**Pros**: Atomic comparison, precise undo  
**Cons**: Manual, per-operation overhead

### Option B: Hourly Automated (Recommended)
```bash
# Add to crontab
0 * * * * cd /Users/home/paperclip && ./scripts/backup-non-git-data.sh ~/.paperclip/hourly-backups && find ~/.paperclip/hourly-backups -mtime +3 -type d -exec rm -rf {} \;
```
**Pros**: Continuous coverage, automatic cleanup  
**Cons**: Disk usage (5.2GB × 24 = 125GB/day if keeping all)

### Option C: Daily + Weekly (Balanced)
```bash
# Crontab:
# Daily at 2 AM
0 2 * * * cd /Users/home/paperclip && ./scripts/backup-non-git-data.sh ~/.paperclip/daily-backups && find ~/.paperclip/daily-backups -mtime +7 -type d -exec rm -rf {} \;

# Weekly Sunday at 3 AM (kept for 30 days)
0 3 * * 0 cd /Users/home/paperclip && ./scripts/backup-non-git-data.sh ~/.paperclip/weekly-backups && find ~/.paperclip/weekly-backups -mtime +30 -type d -exec rm -rf {} \;
```
**Pros**: Good coverage with controlled disk usage  
**Cons**: May miss corruption between backups

## Quick Start

### Backup
```bash
# Single backup
./scripts/backup-non-git-data.sh

# Output: .backups/paperclip-20260501-120000/

# Backup size
du -sh .backups/paperclip-20260501-120000/

# List contents
ls -la .backups/paperclip-20260501-120000/
```

### Housekeeping (Clean Old Backups)

**Preview what will be deleted (dry-run):**
```bash
./scripts/backup-housekeeping.sh --dry-run

# Output:
#  📁 Checking: daily (retention: 7 days)
#    ✓ All backups in daily within retention policy
#  📁 Checking: weekly (retention: 30 days)
#    [DRY-RUN] Would delete: paperclip-20260425-000000 (1.5 GB)
```

**Actually delete old backups:**
```bash
./scripts/backup-housekeeping.sh --execute

# Output:
#  📁 Checking: daily (retention: 7 days)
#    ✓ All backups in daily within retention policy
#  📁 Checking: weekly (retention: 30 days)
#    🗑  Deleted: paperclip-20260425-000000 (1.5 GB)
#    Summary: 1 backup(s) marked for deletion in weekly
```

**Check disk usage after cleanup:**
```bash
du -sh ~/.paperclip/daily-backups ~/.paperclip/weekly-backups/
```

### Restore
```bash
# Restore from specific backup
./scripts/restore-non-git-data.sh .backups/paperclip-20260501-120000

# Verify
ls -la ~/.paperclip/instances/default/secrets/
curl http://localhost:3100/api/health
```

## Restore Procedure (Step-by-Step)

### If Database Corruption Detected

1. **Identify the issue** (check Slack, API errors, activity logs)
   ```bash
   # Last 100 API errors
   curl -s http://localhost:3100/api/activity?status=error | jq '.[] | {actor, action, error}' | head -50
   ```

2. **Locate the backup** (most recent pre-corruption)
   ```bash
   ls -lrt .backups/paperclip-*/ | tail -5
   ```

3. **Stop the service**
   ```bash
   ./paperclip-service.sh pause
   # or: pkill -f paperclip
   ```

4. **Restore**
   ```bash
   ./scripts/restore-non-git-data.sh .backups/paperclip-20260501-120000
   ```

5. **Verify**
   ```bash
   # Check secrets exist
   ls -la ~/.paperclip/instances/default/secrets/master.key
   
   # Check database size is reasonable (should be ~226MB)
   du -sh ~/.paperclip/instances/default/db/
   
   # Start and test
   pnpm dev
   curl http://localhost:3100/api/health
   ```

6. **If still corrupted**, restore to older backup
   ```bash
   ./scripts/restore-non-git-data.sh .backups/paperclip-20260501-080000
   ```

## Detecting Corruption

### Red Flags
- ❌ API returns 500 errors
- ❌ Agents can't read/write documents
- ❌ Companies disappear from dashboard
- ❌ Tasks/issues have nonsensical state
- ❌ Activity logs show unexpected bulk changes
- ❌ `master.key` file deleted or corrupted

### Check Command
```bash
# Test database connectivity
pnpm paperclipai db:health

# List all companies (should be ~6)
curl -s http://localhost:3100/api/companies | jq '.[] | .name' | wc -l

# Check recent activity
curl -s http://localhost:3100/api/activity?limit=50 | jq '.[] | {actor, action, resource, timestamp}'
```

## What NOT to Backup (Git-tracked)

These are already in Git, **don't backup**:
- `server/`, `packages/`, `ui/` — source code
- `doc/` — documentation
- `cli/` — CLI tool
- `tests/` — test suites
- `node_modules/` — dependencies

**To restore source code**, just `git checkout`.

## Security Considerations

### Secrets in Backup
- ✅ `master.key` IS included (needed for recovery)
- ✅ API keys in DB are encrypted with `master.key`
- ⚠️ Store backup locations securely
- ⚠️ Consider encrypting backup files if stored remotely

### Example: GPG Encrypt Backup
```bash
# Encrypt entire backup directory
tar -czf - .backups/paperclip-20260501-120000/ | \
  gpg --symmetric -o backup-encrypted.tar.gz.gpg

# Later, decrypt and restore
gpg -d backup-encrypted.tar.gz.gpg | tar -xz
./scripts/restore-non-git-data.sh extracted/paperclip-20260501-120000
```

## Automation Setup

### Crontab Entry (Daily + Weekly with Housekeeping)

```bash
# Open crontab editor
crontab -e

# Add these lines:

# ═══ DAILY BACKUPS ═══
# Backup at 2 AM daily
0 2 * * * cd /Users/home/paperclip && ./scripts/backup-non-git-data.sh ~/.paperclip/daily-backups >/dev/null 2>&1

# Cleanup old daily backups (keep 7 days) at 2:30 AM
30 2 * * * cd /Users/home/paperclip && ./scripts/backup-housekeeping.sh --execute >/dev/null 2>&1

# ═══ WEEKLY BACKUPS ═══
# Backup at 3 AM on Sundays
0 3 * * 0 cd /Users/home/paperclip && ./scripts/backup-non-git-data.sh ~/.paperclip/weekly-backups >/dev/null 2>&1

# Cleanup old weekly backups (keep 30 days) at 3:30 AM Sundays
30 3 * * 0 cd /Users/home/paperclip && ./scripts/backup-housekeeping.sh --execute >/dev/null 2>&1

# ═══ WEEKLY REPORT (optional) ═══
# Preview cleanup at 4 AM Sundays (email if you want)
# 0 4 * * 0 cd /Users/home/paperclip && ./scripts/backup-housekeeping.sh --dry-run 2>&1 | mail -s "Backup Report" you@example.com
```

Alternatively, copy all lines from `.crontab-entries.txt`:
```bash
cat .crontab-entries.txt | crontab -e
```

### Verify Cron is Running
```bash
# Check crontab
crontab -l

# Monitor cron logs
log stream --predicate 'process == "cron"' --level debug
```

## Troubleshooting

### Backup Fails: Database Still in Use
```bash
# Kill Paperclip processes
pkill -f paperclip
sleep 2

# Retry backup
./scripts/backup-non-git-data.sh
```

### Restore Fails: Insufficient Space
```bash
# Check available space
df -h ~/.paperclip

# If <2 GB free, move old backups
mv ~/.paperclip/daily-backups ~/.backups/archive-$(date +%Y%m%d)

# Try restore again
./scripts/restore-non-git-data.sh .backups/paperclip-20260501-120000
```

### Database Won't Start After Restore
```bash
# Check if PG version matches
ls ~/.paperclip/instances/default/db/PG_VERSION

# If corrupted, delete database and restore
rm -rf ~/.paperclip/instances/default/db
./scripts/restore-non-git-data.sh .backups/paperclip-20260501-120000

# Start fresh
pnpm dev
```

## Summary

| Concern | Solution | Backup Size |
|---------|----------|---|
| Accidental AI corruption | Daily snapshots (retention: 7-14d) | ~1.5 GB per backup |
| Complete data loss | Database + config restore | 5-10 min recovery |
| Key loss (secrets) | Master key stored separately | < 1 sec restore |
| Time to recovery | Pre-identified backup locations | Know backup path |
| Manual overhead | Automated cron task | ~30 min setup once |

**Recommended Setup:**
1. ✅ Daily automated backups (cron) — keep 7 days
2. ✅ Weekly archives — keep 30 days
3. ✅ Quick restore procedure (documented above)
4. ✅ Pre-agent-run backups for critical operations
